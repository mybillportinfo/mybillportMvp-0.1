🔥 Master Prompt: Upgrade Gmail Scanner to Intelligent Bill Extraction

You are a senior full‑stack developer working on MyBillPort, a Canadian bill management app. The current Gmail scanner successfully adds bills, but it fails to extract account numbers and other useful fields. Your task is to upgrade the extraction pipeline to a fintech‑grade intelligent system.

📌 Context

· Tech stack: Next.js (Vercel), Firebase Auth, Firestore, Google Gmail API, Claude API (optional), Node.js.
· Current state:
  · Users connect Gmail via OAuth2 (readonly scope).
  · A scan endpoint fetches recent bill‑related emails (subject containing "bill", "invoice", "due").
  · It passes email content to Claude with a basic prompt and stores extracted billerName, amount, dueDate.
  · Problem: Account number is missing; other fields (minimum payment, statement date) are ignored.
· Goal: Build a robust extraction pipeline that:
  · Extracts all relevant fields (see schema below)
  · Uses regex for deterministic extraction where possible
  · Falls back to AI for ambiguous cases
  · Computes a confidence score
  · Enriches with biller metadata from a mapping table

---

🏗️ New Extraction Pipeline Architecture

You will implement the following flow inside the existing /api/gmail/scan endpoint (or in a helper module). The code must be modular, well‑commented, and production‑ready.

```
[Fetch emails] → [Pre‑process] → [Regex extraction] → [AI fallback] → [Confidence scoring] → [Biller enrichment] → [Return to frontend]
```

📦 Data Schema for Extracted Bill

Create a TypeScript interface and ensure the final output conforms to:

```typescript
interface ExtractedBill {
  // Core
  billerName: string;
  billerDomain: string;        // from sender email (e.g., "enbridge.com")
  accountNumber: string | null; // cleaned, no spaces/dashes
  amountDue: number;
  dueDate: string;              // ISO date (YYYY-MM-DD)
  statementDate: string | null; // ISO date
  minimumPayment: number | null;
  totalBalance: number | null;
  currency: string;             // default "CAD"

  // Enrichment
  confidenceScore: number;      // 0–100
  detectionMethod: 'regex' | 'ai' | 'hybrid';

  // Metadata
  sourceEmailId: string;
  sourceThreadId: string;
  hasAttachment: boolean;
  attachmentFilenames: string[];
  extractedAt: Date;
}
```

🧩 Detailed Implementation Tasks

1. Pre‑processing

· Use mailparser to extract plain text from HTML emails.
· If email has PDF attachments, use pdf-parse to extract text (store extracted text separately).
· Normalize text: collapse multiple spaces, trim lines, remove excessive line breaks.

2. Regex Extraction (Deterministic)

Create a module extractWithRegex.js that exports a function tryRegexExtract(emailText). Implement patterns for:

Account number:

· /Account\s*[:#]?\s*([A-Z0-9\s-]{5,30})/i
· /(?:Acct|Account Number)[\s:]*([A-Z09\s-]+)/i
· Clean result: remove spaces/dashes, keep alphanumeric.

Amount due:

· /(?:Amount due|New Charges|Current Bill|Payment Due)\s*[:$]?\s*\$?([\d,]+\.\d{2})/i
· /\$([\d,]+\.\d{2})\s*(?:due|by|on)/i

Due date:

· /Due\s*date\s*[:#]?\s*([A-Za-z]+\s+\d{1,2},?\s*\d{4})/i
· /Payment\s*due\s*[:#]?\s*(\d{1,2}[-\/]\d{1,2}[-\/]\d{2,4})/i
· Convert to YYYY-MM-DD.

Statement date:

· /Statement\s*date\s*[:#]?\s*([A-Za-z]+\s+\d{1,2},?\s*\d{4})/i

Minimum payment:

· /Minimum\s*payment\s*[:#]?\s*\$?([\d,]+\.\d{2})/i

Total balance:

· /Total\s*balance\s*[:#]?\s*\$?([\d,]+\.\d{2})/i

Return an object with found fields (null if not found).

3. AI Extraction (Fallback)

If regex misses any required fields (especially account number), call Claude with a structured prompt. Create a function extractWithAI(emailText, regexHints).

Prompt:

```
You are an expert at extracting billing information from emails. Given the email content below, return a JSON object with these exact fields:

- billerName: the company name (e.g., "Enbridge", "Rogers")
- accountNumber: the customer's account number (may contain spaces, dashes; clean if possible)
- amountDue: the current bill amount as a number
- dueDate: the date payment is due (YYYY-MM-DD)
- statementDate: the date the statement was issued (YYYY-MM-DD, if available)
- minimumPayment: the minimum amount due (if present)
- totalBalance: the total account balance (if different from amountDue)
- currency: "CAD" unless otherwise specified

Rules:
- If a field is not present, use null.
- Prefer amounts labeled "amount due" or "new charges" over "balance".
- Account numbers may be partially masked (e.g., ****1234) – extract the unmasked part if visible.
- Return ONLY valid JSON, no extra text.

Email content:
${emailText}
```

Parse the response and merge with regex results (prefer regex for fields it found, use AI for missing ones).

4. Confidence Scoring

Write a function computeConfidence(extracted, regexResults, aiResults) that returns a score 0–100 based on:

Field Weight Condition
amountDue 40% Found by regex → 40, by AI → 30, else 0
dueDate 30% Found by regex → 30, by AI → 20, else 0
accountNumber 20% Found by regex → 20, by AI → 15, else 0
billerName 10% Matches known domain → 10, by AI → 8, else 0

Add bonus if multiple emails in same thread confirm same data.

5. Biller Recognition Engine

Create a collection in Firestore billerMapping with documents:

```json
{
  "domain": "enbridge.com",
  "canonicalName": "Enbridge",
  "category": "gas",
  "paymentUrl": "https://enbridge.com/pay",
  "confidence": 1.0
}
```

In extraction, look up sender email domain. If found, set billerName to canonicalName (overriding AI). If not, use AI result.

6. Integration into Scan Endpoint

Modify /api/gmail/scan to:

· For each email, run the full extraction pipeline.
· Store the resulting ExtractedBill objects in an array.
· Return the array to the frontend (for user review).
· Also store them temporarily in Firestore under users/{userId}/pendingBills with status: 'pending'.

7. Additional Enhancements

· Gmail labeling: After successful extraction, apply a label BillPort/Imported (create if not exists) to the email to prevent reprocessing.
· Duplicate detection: Check if same sourceEmailId already processed; skip if yes.
· Error handling: Wrap each step in try/catch, log errors, continue with next email.
