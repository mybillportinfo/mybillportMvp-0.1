🔥 MASTER PROMPT: Gmail Auto Bill Detection for MyBillPort

You are a senior full‑stack developer. Generate production‑ready code for the Gmail integration feature of MyBillPort, a Canadian bill management app.

Tech Stack:

· Frontend: Next.js (Vercel)
· Backend: Next.js API routes (Node.js)
· Authentication: Firebase Auth
· Database: Firestore
· External API: Google Gmail API (readonly)
· OAuth2: Google OAuth with gmail.readonly scope
· Deployment: Code pushed to GitHub, deployed on Vercel

Goal: Allow users to connect their Gmail account so the app can automatically scan for bill emails and extract bill data (merchant, amount, due date). Store tokens securely, handle refresh, and provide endpoints for connect, callback, scan, and disconnect.

---

📁 API Routes to Implement

All routes must verify the Firebase ID token (from Authorization header) and only allow authenticated users.

1. POST /api/gmail/connect

· Purpose: Generate Google OAuth URL and return it to the frontend.
· Logic:
  · Use Google OAuth2 client with client ID, secret, and redirect URI (configured in environment variables).
  · Scopes: https://www.googleapis.com/auth/gmail.readonly
  · Generate a random state parameter (store temporarily in Firestore or session).
  · Return { url } to frontend.
· Security: Must include Firebase auth check.

2. GET /api/gmail/callback

· Purpose: Handle OAuth callback from Google.
· Logic:
  · Receive code and state from query params.
  · Verify state matches stored value (to prevent CSRF).
  · Exchange code for tokens using Google OAuth2 client.
  · Store tokens in Firestore under users/{userId}/gmailIntegration:
    ```json
    {
      access_token: "string",
      refresh_token: "string",
      expiry_date: number (timestamp in ms),
      lastScanDate: null | timestamp,
      createdAt: timestamp,
      updatedAt: timestamp
    }
    ```
  · Redirect user to frontend success page (e.g., /settings?gmail=connected).
· Security: Firebase auth check not needed here (OAuth flow initiated from frontend, but we can rely on state to link to user). After exchange, we can identify user from state.

3. POST /api/gmail/scan

· Purpose: Trigger a scan of the user's Gmail inbox for bill emails.
· Logic:
  · Fetch the user's Gmail tokens from Firestore.
  · If token expired, refresh it automatically (use refresh token).
  · Use Gmail API to search for messages with query:
    ```
    subject:(bill OR invoice OR due OR statement) newer_than:90d
    ```
  · For each message, fetch metadata (sender, subject, snippet) and process:
    · Extract billerName from sender (email address or display name).
    · Extract amount from snippet or email body (use regex patterns).
    · Extract dueDate from snippet or body (common date patterns).
    · Do not store full email body – only extracted data.
  · Return array of extracted bill candidates to frontend (for user confirmation).
  · Update lastScanDate in Firestore.
  · For subsequent scans, use newer_than:7d (based on lastScanDate).
· Error handling: Handle expired token, invalid token, Gmail API quota errors.

4. POST /api/gmail/disconnect

· Purpose: Disconnect Gmail integration and revoke tokens.
· Logic:
  · Retrieve tokens from Firestore.
  · Call Google's revoke endpoint: https://oauth2.googleapis.com/revoke?token={access_token}.
  · Delete the gmailIntegration document from Firestore.
  · Return success.

🔐 Security & Configuration

· Store Google OAuth credentials in Vercel environment variables:
  · GOOGLE_CLIENT_ID
  · GOOGLE_CLIENT_SECRET
  · GOOGLE_REDIRECT_URI (e.g., https://mybillport.com/api/gmail/callback)
· Use Firebase Admin SDK with service account (store in FIREBASE_SERVICE_ACCOUNT as JSON string).
· All tokens must be stored encrypted in Firestore (at least use Firestore security rules to restrict access to the user's own document).
· Implement rate limiting to avoid abuse (optional but recommended).
