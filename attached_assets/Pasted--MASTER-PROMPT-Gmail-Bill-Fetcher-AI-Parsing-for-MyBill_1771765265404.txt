🔥 MASTER PROMPT: Gmail Bill Fetcher + AI Parsing for MyBillPort

You are a senior full‑stack engineer (Next.js, Firebase, TypeScript). Your task is to build a feature that lets users connect their Gmail account to automatically discover and parse bill emails using AI, then present them for confirmation. This will be a zero‑cost alternative to Plaid, using free tiers of Gmail API and Claude (with minimal usage).

📌 Context

· App: MyBillPort – Next.js (Pages Router), TypeScript, Tailwind CSS, Firebase (Auth, Firestore, Cloud Functions).
· Goal: Users click “Connect Gmail” → OAuth consent → background sync of emails from known billers → AI extracts amount, due date, merchant → user reviews and adds to bills.

🧱 High‑Level Architecture

```
[Frontend] – Button to connect Gmail (OAuth)
       │
       ▼
[Firebase Auth] – Sign in with Google (scopes: gmail.readonly)
       │ (stores refresh token in Firestore)
       ▼
[Cloud Function] – Scheduled (e.g., daily) for each user with token
       │ – Calls Gmail API to fetch recent emails from billers
       │ – For each email, extracts text/content + attachments
       │ – Calls Claude API to parse into structured bill data
       │ – Stores parsed data in Firestore `pendingBills` collection
       ▼
[Frontend] – Pending Bills list page: user reviews and confirms (→ adds to `bills`) or rejects.
```

📦 Detailed Requirements

1. Gmail OAuth & Token Storage

· Frontend: Add a “Connect Gmail” button on the user settings page.
· Use Firebase Auth’s Google sign‑in provider with custom scopes:
  ```typescript
  const provider = new GoogleAuthProvider();
  provider.addScope('https://www.googleapis.com/auth/gmail.readonly');
  ```
· After sign‑in, the OAuth tokens (especially the refresh token) are not automatically available in Firebase. You must use the signInWithPopup result to get the OAuthCredential and extract the access token and refresh token.
· Backend: Store the refresh token securely in Firestore (e.g., users/{userId}/gmailTokens). Encrypt it if possible (but for MVP, at least store in a secure collection with strict security rules).
· Token refresh: Implement a helper that uses the refresh token to obtain a new access token when needed (using Google’s OAuth endpoint).

2. Cloud Function: Scheduled Email Fetch

· Create a Firebase Cloud Function (region: northamerica-northeast1) triggered via Pub/Sub (e.g., once per day).
· Query all users who have stored Gmail tokens.
· For each user:
  · Refresh the access token.
  · Call Gmail API users.messages.list with query: from: (torontohydro OR rogers OR bell OR enbridge OR ...) – start with a hardcoded list of Canadian billers, but eventually make it user‑configurable.
  · For each message, fetch full content (format: 'full').
  · Extract plain text parts (and attachment IDs). If attachment is PDF, download it and convert to base64.
· Gmail API quota: Free tier allows 1,000,000 requests/day; you’re safe.

3. AI Parsing with Claude (or Gemini)

· For each email (or attachment), prepare a prompt for Claude.
· Use Claude API (model claude-3-sonnet-20240229) to extract:
  · merchant name
  · total amount (as number)
  · due date (as YYYY-MM-DD)
  · account number (if present)
  · confidence (high/medium/low)
· Prompt example:
  ```
  You are a bill parser. Extract the following from this email content:
  - merchant (the company name)
  - total amount due (numeric, without currency symbol)
  - due date (in YYYY-MM-DD format)
  - account number (if present, else null)
  - confidence (high/medium/low)
  
  Return ONLY a JSON object. If a field is missing, use null.
  
  Email content:
  ${emailText}
  ```
· If the email includes a PDF attachment, convert it to text (using pdf-parse library) and send that text to Claude. Alternatively, Claude API can accept PDFs as base64 if you use the anthropic-beta: pdfs-2024-09-25 header (check latest docs). For simplicity, start with text extraction.
· Cost: Claude Sonnet is cheap (~$3/million tokens). A typical bill email is under 1000 tokens → ~$0.003 per scan.

4. Store Parsed Bills in Pending Collection

· In Firestore, create a pendingBills subcollection under each user: users/{userId}/pendingBills/{docId}.
· Each document should contain:
  · gmailMessageId (to avoid duplicates)
  · merchantName, amount, dueDate, accountNumber, confidence
  · rawEmailSnippet (optional, for debugging)
  · status: 'pending'
  · createdAt timestamp
· After parsing, check if a bill with same gmailMessageId already exists to prevent duplicates.

5. Frontend: Pending Bills Review Page

· Create a page /pending-bills that lists all documents from pendingBills with status: 'pending'.
· For each, display a card with:
  · Merchant, amount, due date, account (if any), confidence badge.
  · Buttons: “Confirm” and “Reject”.
· On Confirm:
  · Create a new bill in the bills collection (using your existing addBill logic).
  · Update the pending bill status to confirmed (or delete it).
· On Reject: update status to rejected (or delete).
· Add a “Refresh” button to manually trigger the sync (maybe call a callable function that runs the fetch immediately).

6. Security & Privacy

· Firestore rules: ensure users can only read/write their own gmailTokens and pendingBills.
· Gmail tokens: treat them as secrets. Use Firebase security rules to restrict access.
· Inform users via UI that you will only read emails from known billers and only to extract bill data.
