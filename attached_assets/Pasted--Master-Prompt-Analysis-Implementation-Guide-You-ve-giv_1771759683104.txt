🧩 1. AI Bill Insights Button

Goal

Add a button on each bill card and dashboard summary that, when clicked, shows trend analysis, percentage change, cost breakdown, and an intelligent explanation.

Technical Approach

· Frontend:
  · Add a button (e.g., "AI Insights") next to each bill in the list.
  · On click, open a modal or expandable section with the generated insights.
  · Use a loading state while fetching/calculating.
· Backend / Logic:
  · If you have an OpenAI API key, you can call a Vercel API route /api/insights that:
    1. Fetches historical data for that bill (from Firestore).
    2. Formats a prompt with the data.
    3. Calls GPT to generate a narrative and structured insights.
    4. Returns JSON.
  · Fallback (no API key): Use deterministic rules:
    · Calculate month-over-month change, average, min, max.
    · Generate simple text like "Your internet bill has increased 5% over the last 3 months."
  · Store insights in Firestore? Not necessary; generate on demand.
· Data Required:
  · All bills for the same billerName belonging to the user, sorted by due date.
  · Ensure dueDate is stored as a timestamp.

Implementation Steps

1. Create a new Vercel API route pages/api/insights.ts that accepts userId and billerName (or bill ID) and returns insights.
2. In the frontend, on button click, call this API, show a loading spinner, and display results in a modal.
3. If no historical data (less than 2 bills), show a friendly message.

📈 2. Spike Detection

Goal

Automatically detect unusual increases or decreases (>20% vs previous month or 3‑month average) and show a badge.

Technical Approach

· Trigger: After adding or editing a bill, run a calculation.
· Logic:
  · Get the latest bill amount and the previous bill amount (or average of last 3).
  · Calculate percentage change.
  · If >20% increase → mark as unusual_increase.
  · If >20% decrease → mark as unusual_drop.
  · Store this flag in the bill document (e.g., spikeStatus field).
· Frontend:
  · When displaying a bill, if spikeStatus exists, show a colored badge with explanation.
· Performance:
  · Use a Firestore trigger (Cloud Function) to update spikeStatus automatically when a bill is created/updated.
  · Alternatively, do it client-side after save (simpler but less robust if user has multiple devices).

Implementation Steps

1. Create a Cloud Function detectSpike that listens to onWrite of bills, computes spike, and updates the bill doc.
2. In the bill card component, conditionally render a badge based on spikeStatus.
3. Add a tooltip with explanation (e.g., "20% higher than last month").

📊 3. Annual Cost Projection

Goal

Show projected yearly spending per bill and total across all bills, factoring in trends.

Technical Approach

· Calculation:
  · For each bill, compute average monthly amount (or use trend-adjusted projection).
  · Multiply by 12 for annual projection.
  · Sum across all bills for total.
· Trend Adjustment (optional):
  · If a bill is consistently increasing, you could project a higher amount (e.g., fit a linear regression).
  · Keep it simple first: use average of last 3 months.
· Frontend:
  · Add a new card on dashboard titled "Annual Cost Projection".
  · Inside, list each bill with its projected annual cost, and a total at the bottom.
  · Optionally, use a simple chart library (like Chart.js or Recharts) to show a bar chart of categories.
· Data Source:
  · All bills for the user, grouped by billerName (or category).
  · Need to handle recurring bills properly (those with recurring: true).

Implementation Steps

1. Create a utility function calculateAnnualProjection(bills) that returns per-bill and total.
2. Use it in the dashboard (client-side or via a Cloud Function for caching).
3. Display results in a clean card.

💰 4. Savings Opportunity Score

Goal

Generate a 0–100 score based on bill stability and optimization, with explanation.

Technical Approach

· Scoring Factors:
  · Volatility: standard deviation of amounts across time (lower volatility → higher score).
  · Spike count: number of spikes in the last 6 months.
  · Category spending vs. benchmark (if you have benchmarks, e.g., average Canadian internet bill).
  · Recurring vs. irregular: recurring bills are easier to manage.
· Score Logic (example):
  · Start at 100.
  · Subtract points for each spike (e.g., -5 per spike).
  · Subtract points if average amount > benchmark (e.g., -10 if >20% above).
  · Add points if no changes in the last 3 months.
· Frontend:
  · Display a gauge or progress bar with the score.
  · Add a label: "Optimized", "Moderate", or "Needs Attention".
  · Show a short explanation (e.g., "Your bills are stable, but your internet is 15% above average").
· Implementation:
  · Calculate on the fly (client-side) from bills data.
  · Cache if expensive.

Implementation Steps

1. Create a function calculateSavingsScore(bills) that returns score, label, and explanation.
2. Call it on the dashboard after loading bills.
3. Render the score with styling.

🧱 Overall Architecture Considerations

Feature Where to Compute Data Dependency Complexity
AI Insights Vercel API (or client) Historical bills, optional OpenAI Medium
Spike Detection Cloud Function (Firestore trigger) Bill amounts, dates Low
Annual Projection Client (or Cloud Function) All bills Low
Savings Score Client All bills, maybe benchmarks Medium

Potential Issues & Mitigations

1. Performance:
   · If a user has hundreds of bills, computing projections and scores on the client may be heavy. Consider caching results in Firestore (e.g., a userStats document updated via Cloud Function after each bill change).
2. OpenAI Costs:
   · The "AI Insights" feature could rack up costs if users click it often. Implement a rate limit (e.g., once per bill per session) or use a simple fallback.
3. Data Consistency:
   · Spike detection via Cloud Function ensures all devices see the same flag. Use Firestore triggers.
4. User Experience:
   · Add loading skeletons for insights and projections.
   · Ensure mobile responsiveness.
